Avaliação Técnica — Decisões, Trade-offs e Evidências

Resumo da solução para apresentação (nível Tech Lead).

1) Escopo e Requisitos

O que foi pedido
  • API que retorna dados (read-only), sem banco de dados.
  • Persistência via JSON (ou CSV).
  • Endpoints típicos de página de item (detalhe, recomendações, busca).
Como atendi
  • FileProductRepository lendo data/*.json do classpath.
  • Arquitetura Hexagonal (ports/adapters) com domínio isolado.
  • Read-only + Observabilidade (Actuator/Prometheus) + OpenAPI/Swagger.

2) Arquitetura

Camadas
  • domain: entidades/VO e ProductRepository (porta).
  • application: casos de uso (GetProductDetail, SearchProducts, GetRecommendations).
  • infrastructure: FileProductRepository (JSON + índice + cache).
  • app: Spring Boot (controllers, config, docs, health/metrics).
Por que
  • Permite trocar storage (CSV/HTTP/DB) sem tocar domínio/use-cases.
  • Controllers não conhecem infra → baixo acoplamento, testes simples.

3) Dados e Índices

Leitura
  • Jackson desserializa JSON em mapas e VO.
  • Índices em memória: tokens de título, categoria, marca.
Busca
  • Tokenização simples (lowercase + split regex).
  • Pagina: limit/offset, total calculado.
  • Recomendações: união de {mesma categoria, mesma marca} menos o próprio item.

4) Performance & Cache

Hot paths
  • Detalhe por ID cacheado via Caffeine (TTL configurável).
  • Índices evitam varredura O(n) nas consultas.
HTTP caching (bônus sugerido)
  • ETag/Last-Modified para /products/{id}.
  • Cache-Control conservador no edge/CDN.

5) Observabilidade & Operação

Actuator
  • /actuator/health com probes.
  • /actuator/prometheus para scraping.
Métricas & Logs
  • Latência/contagem por rota via Micrometer (fácil de estender).
  • Logs estruturados prontos para ELK/OTel.

6) Contratos & DX

OpenAPI
  • Swagger UI em /docs.
  • DTO/VO estáveis; Bean Validation em entradas.
Erros
  • 404 para ID inexistente.
  • 400 para parâmetros inválidos (@Min/@Max etc.).

7) Testes

Integração
  • ApiIT (MockMvc) cobrindo rotas principais.
Extensões sugeridas
  • Contract tests contra OpenAPI.
  • Unidades para índice/busca e recomendações.

8) ADRs (curtos)

  1. ADR-001 — Arquitetura Hexagonal: separar domínio de infra por evolutividade.
  2. ADR-002 — Storage JSON: aderência ao enunciado; sem DB real.
  3. ADR-003 — Índice em Memória: tempo de inicialização × tempo de consulta; reduz O(n).
  4. ADR-004 — Cache TTL: Caffeine no detalhe/ID para reduzir I/O.
  5. ADR-005 — Paginação limit/offset: simples; alternativa futura: cursor/slice.
  6. ADR-006 — Recomendações heurísticas: por categoria/marca; fácil evoluir.
  7. ADR-007 — Observabilidade: Actuator + Prometheus para SLO e troubleshoot.
  8. ADR-008 — Read-only: sem endpoints de escrita; evita overscope.

9) Roadmap (futuro)

  • ETag/If-None-Match e Cache-Control.
  • Admin: reload do dataset + warmup do cache.
  • Adapter CSV alternativo.
  • Recs com sinais (popularidade, preço, afinidade de marca).
